# A Theory of Dusk

## What it models

Dusk answers a narrow class of everyday astronomical questions asked from a point on Earth's surface at a calendar day: when does the Sun rise, transit, and set; when does twilight begin and end at three standard depressions below the horizon; when does the Moon rise and set; what does the Moon look like right now. The world it models has two moving bodies (Sun, Moon), a single stationary viewpoint (the Observer), and a clock that is simultaneously two things — a civil local day (bounded by the observer's wall-clock midnights) and a continuous astronomical instant (a Julian date in UTC). Almost every subtlety in the code comes from keeping those two clocks from contaminating each other.

The primary domain nouns are `Observer`, `SunEvent`, `MoonEvent`, `TwilightEvent`, and `LunarPhaseInfo`. Internally, there is also a smaller vocabulary of coordinate frames — `equatorial`, `horizontal`, `ecliptic` — that the public API deliberately hides. A new maintainer should read those internal types as scratch-paper shapes for Meeus's algorithms rather than as part of the library's domain model.

## The load-bearing ideas

**The Observer is a validated capability, not a struct.** In v3 the fields were unexported and `NewObserver` became the only legal way to make one. The `validObserver` check at the top of every public function tests `obs.loc == nil` specifically because a zero-value `Observer` is the one way a caller can get an invalid instance past the type system. Every subsequent calculation trusts the coordinates are finite and in range. This "validate once at the boundary" pattern is the single most important convention: any new public function must begin with `validObserver(obs)` and `validJulianDateRange(date)`, or the safety contract breaks.

**Two absences, two different meanings.** A zero-value `time.Time` in a result means "this event did not occur on this specific day, but nothing is wrong" — the Moon simply didn't rise before midnight, or set after. A sentinel error (`ErrCircumpolar` / `ErrNeverRises`) means "the geometry makes the event impossible at this latitude on this day." This distinction is load-bearing: callers are expected to branch on `.IsZero()` for the first case and `errors.Is` for the second. `MoonEvent.AboveHorizon` exists precisely to disambiguate the ambiguous case of a Moon that is silently above the horizon all day from one that is silently below.

**Degrees everywhere, normalized at the leaves.** The `trig.go` layer is tiny but absolute: there is no raw `math.Sin` in the rest of the package, and `mod360` / `mod24` are used so aggressively that angle wraparound is not something higher layers ever think about. `clamp` inside `asinx`/`acosx` is a deliberate trade — it silently hides upstream bugs where a computed cosine drifts to 1.0000001, because the authors decided that regression against Meeus and USNO reference data is a better correctness check than runtime NaN panics. If you change `clamp`, you are changing the correctness discipline of the whole library.

**The "6-step solar parameter sequence."** `computeSolarParams` exists because `SunriseSunset` and `twilight` both need the same six-step chain (mean anomaly → equation of center → ecliptic longitude → declination → transit JD) and earlier versions duplicated it three ways. Everything downstream of `solarParams{delta, jTransit}` is a pure function of that pair plus the observer's latitude and a depression angle. Treat this as the chokepoint: a new Sun-based boundary (e.g., "blue hour") is a new depression value passed to `solarHourAngle`, not a new algorithm.

**Two different clock conventions, deliberately kept asymmetric.** Solar routines round the Julian day to an integer (NOAA's method), giving a sub-minute answer that is good enough for sunrise/sunset and twilight. Lunar routines use continuous Julian centuries and the full Meeus Chapter 47 periodic-term tables (100+ coefficients). The asymmetry reaches deep: `solarDeclination` uses _mean_ obliquity only, while `eclipticToEquatorial` applies full nutation (both Δψ and Δε). The CLAUDE.md note flags this explicitly as "intentional asymmetry" — the simplified sun path is NOAA-faithful, the lunar path is Meeus-faithful, and trying to unify them would make one of them worse.

**GMST has a midnight-vs-instant split that looks wrong and isn't.** In `greenwichMeanSiderealTime`, `T` is computed from midnight UTC of the date while the linear `360.98564…×(JD−J2000)` term uses the full `JD`. There is a comment forbidding "simplification." This is the single most likely place a well-meaning refactorer would introduce a subtle error.

## The seams

**Calendar day vs. astronomical instant.** The thinnest seam in the system is the conversion from an input `time.Time` to "the local day this call is about." The history of commit 2719beb shows this was gotten wrong once already: `SunriseSunset` and `twilight` used to extract the date from the input's own timezone, so a caller passing UTC midnight for a California observer got the previous day. The current convention — `date.In(obs.loc)` first, then reconstruct midnight — is now consistent across all three of `SunriseSunset`, `twilight`, and `MoonriseMoonset`. `LunarPhase` is the exception: it takes an instant, not a day, and has no `Observer`, because phase is a property of the Sun-Earth-Moon geometry and the observer is irrelevant. A maintainer adding any new public function must decide on day one which of these two regimes it belongs to.

**DST handling lives inside `MoonriseMoonset`.** The line `time.Date(..., obs.loc).UTC()` constructs local midnight, then converts to UTC, so the scan length (`scanMinutes`) is 23 × 60 on spring-forward days and 25 × 60 on fall-back days. This is why the loop computes `scanMinutes` dynamically rather than hard-coding 1440. Any rewrite of moonrise detection must preserve this property.

**The Julian-date range is defined by int64 nanoseconds, not by astronomy.** `julianDate` uses `UnixNano()`, which silently returns 0 outside its range. `validJulianDateRange` is how the library refuses to compute garbage — `julianDateMin`/`julianDateMax` are literally `time.Unix(0, math.MinInt64)` and its complement, giving roughly 1677–2262. This is a pure implementation-driven boundary that has leaked into the public contract as `ErrDateOutOfRange`. If someone ever replaced the Unix-nano path with a direct calendar-to-JD formula, the public range would widen and `ErrDateOutOfRange` would become misleading.

**The one-minute scan is a contract, not an optimization.** `MoonriseMoonset` iterates minute-by-minute (~1440 evaluations, 1-2 ms) because the Moon's angular motion and the minute-by-minute altitude sign change are easier to reason about than solving a transcendental equation. The doc comment makes the cost visible so callers computing month-long calendars can plan for it. Events shorter than a minute can be missed — acknowledged, not bugged. Replacing this with a bisection or analytic solver would be a theory change, not a performance tweak.

**Twilight is all-or-nothing across two days.** The `twilight()` helper computes tonight's dusk from today's `computeSolarParams` and tomorrow morning's dawn from _tomorrow's_. Either side failing (e.g., at 75°N the polar transition happens between Nov 25 and Nov 26) aborts the entire call. The doc comment explicitly tells callers who want partial results to compute each boundary themselves. This is a principled seam: a `TwilightEvent` without both ends is not a meaningful value in this vocabulary.

## What the shape accommodates, and what it resists

Easy changes: **new solar depression boundaries** (a "blue hour" at 4° or a "golden hour" at −6° wrapper reusing `twilight()`); **new result fields** on existing event types, since everything is produced by a single code path per event; **new Meeus coefficient rows** — the lunar tables are declared as `[...]lunarLongDistCoeff{...}` literals specifically so that pasting a row works; **tighter accuracy** in solar calculations, by migrating `SunriseSunset` from NOAA-rounded to Meeus-continuous the same way the lunar side already is.

Hard changes: **anything that requires elevation** — it was deliberately removed in v3 on the grounds that the ~0.5' correction wasn't worth the complexity. Re-adding it means reintroducing a field to `Observer`, a parameter to `solarHourAngle`, and updating every test expectation. **Any object other than Sun or Moon** — there is no planet-neutral "rise/set for given equatorial coordinates" function anymore; v2 had one (`ObjectTransit`), v3 removed it as part of the API shrink. Adding planets back means either resurrecting that abstraction or duplicating the minute-by-minute scan with a different position function. **A precise moonrise** would require a coordinate solver and an interpolation scheme, breaking the "scan-until-sign-change" mental model.

Where a maintainer would look first for a given request depends on knowing the theory: a timezone bug → start at the `date.In(obs.loc)` line at the top of the affected public function. A wrong rise time at high latitude → `solarHourAngle`'s `cosHA < -1` / `> 1` branches and whether the depression angle is 0. A Moon test failing after a library update → whether `clamp` is hiding something in `asinx`/`acosx`. Wrong phase name → `lunarPhaseName`'s 22.5°-boundary cutoffs, not the phase-angle math.

A maintainer who didn't hold the theory would be most likely to damage things by: (a) "simplifying" `greenwichMeanSiderealTime` to use the instant for both terms; (b) making `solarDeclination` apply full nutation "for consistency"; (c) removing `clamp` after diagnosing a masked bug and not restoring the discipline it enforces; (d) reading `SunriseSunset`'s rounded `J` and `MoonriseMoonset`'s continuous Julian day as inconsistent and aligning them; (e) replacing `UnixNano`-based Julian dates to widen range without updating the `ErrDateOutOfRange` surface.

## Uncertainties and tensions

I am inferring from code alone (not from authors) a few things worth flagging:

- I read the "NOAA for the Sun, full Meeus for the Moon" split as an intentional accuracy/simplicity trade, because CLAUDE.md calls it out — but the library would be more internally consistent if solar also used continuous JD, and I can't tell whether the asymmetry was preserved for fidelity to NOAA's widely-cross-checked method or just because no one has unified it yet.
- `solarMeanAnomaly(J)` takes days since J2000 while `solarMeanAnomalyFromCentury(T)` takes centuries; they're both present internally because the solar pipeline wants days and the lunar pipeline wants centuries. CLAUDE.md flags `solarMeanAnomaly` as the "odd one out" that takes days. This is a real tension — one function in two unit systems — and if a third caller arrives it will push toward consolidation.
- `hourAngle` takes RA in degrees but LST in hours, and the doc comment loudly warns callers not to pre-convert. This is the sort of mixed-unit API that usually signals an incomplete refactor; my guess is that it survives because inlining the `lst*15` conversion everywhere would obscure the Meeus equations. I'd treat it as drift that is stable for now, not a principled seam.
- `MoonriseMoonset` scans the full day even after finding both rise and set (it only `break`s when both are set), which is fine, but the early-exit makes the worst-case cost depend on event ordering and latitude. This is more of a performance note than a theory gap.
- The `errString` constant-error trick is elegant but orthogonal to the astronomy; a reader coming from a more typical Go codebase might waste time wondering why errors aren't `errors.New`. The reason is purely "I want these comparable by `==` and unassignable."
- The v2→v3 API shrink removed a lot of previously exported helpers (`SolarPosition`, `LunarPosition`, `EclipticToEquatorial`, etc.). They still exist as unexported functions. If a future maintainer wants to support "give me the Sun's RA/Dec at an instant" without adding a whole new event type, the machinery is right there — but the theory says the library is event-oriented, not position-oriented, and that boundary was drawn on purpose.

---

The theory of this codebase, stated as compactly as possible: _dusk computes a small set of day-scoped observer events for two bodies using two deliberately different fidelity regimes, with the civil day and the astronomical instant kept rigorously distinct, validation pushed to construction time, absences encoded two different ways to preserve meaning, and all angle arithmetic performed in degrees through a thin trig veneer whose clamping is load-bearing._
